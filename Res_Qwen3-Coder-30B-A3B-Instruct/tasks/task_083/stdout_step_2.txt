		<title>GitHub for Beginners: Getting started with Git and GitHub in VS Code</title>
		<link>https://github.blog/developer-skills/github/github-for-beginners-getting-started-with-git-and-github-in-vs-code/</link>
		
		<dc:creator><![CDATA[Natalie Guevara]]></dc:creator>
		<pubDate>Mon, 25 May 2026 16:00:00 +0000</pubDate>
				<category><![CDATA[Developer skills]]></category>
		<category><![CDATA[GitHub]]></category>
		<category><![CDATA[Git]]></category>
		<category><![CDATA[Github]]></category>

					<description><![CDATA[<p>Discover how to use VS Code to interact with GitHub and maintain your projects.</p>
<p>The post <a href="https://github.blog/developer-skills/github/github-for-beginners-getting-started-with-git-and-github-in-vs-code/">GitHub for Beginners: Getting started with Git and GitHub in VS Code</a> appeared first on <a href="https://github.blog">The GitHub Blog</a>.</p>
]]></description>
										<content:encoded><![CDATA[<!DOCTYPE html PUBLIC "-//W3C//DTD HTML 4.0 Transitional//EN" "http://www.w3.org/TR/REC-html40/loose.dtd">
<html><body><p>Welcome back to GitHub for Beginners. We&rsquo;ve covered a lot this season, so make sure to check out our other episodes. <a href="https://github.blog/developer-skills/github/github-for-beginners-getting-started-with-oss-contributions/">Our most recent one</a> was all about open source, what it is and how to contribute to the community.</p>



<p>This time, we&rsquo;re going to take a look at VS Code, a free popular source code editor provided by Microsoft. It has a fair amount of functionality built in that integrates with GitHub, which is what we&rsquo;ll be taking a look at today. Using GitHub in VS Code reduces context switching, streamlines your workflow, and boosts your productivity. By the end of this post, you&rsquo;ll understand how to use VS Code to initialize a repository, switch branches, as well as stage, commit, and push your changes. And the best part is, you&rsquo;ll be able to do all this without leaving the editor.</p>



<p>Note that if you want to follow along with this blog post (or the video), you will need to install both <a href="https://git-scm.com/">Git</a> and <a href="https://code.visualstudio.com/download">VS Code</a>. If you need a refresher on how to install Git, you can check out one of our <a href="https://github.blog/developer-skills/programming-languages-and-frameworks/what-is-git-our-beginners-guide-to-version-control/">earlier GitHub for Beginners episodes</a>.</p>



<p>As always, if you prefer to watch the video or want to reference it, we have all of our <a href="https://gh.io/gfb">GitHub for Beginners episodes available on YouTube</a>.</p>



<h2 class="wp-block-heading" id="h-first-some-basics">First some basics</h2>



<p>You probably already know that GitHub is a resource that hosts only copies of your code in repositories. So what is Git? Git is the program for managing that source code, and it can be used in multiple different ways (e.g., from the command line, through VS Code, etc.). Visual Studio Code, often abbreviated as VS Code, leverages Git to enable you to manage your code in GitHub.</p>



<h2 class="wp-block-heading" id="h-initializing-a-folder">Initializing a folder</h2>



<p>The first step to using Git with VS Code is initializing a folder to reflect your repository on GitHub.</p>



<ol class="wp-block-list">
<li>Open VS Code.</li>



<li>Select the top icon (the <strong>Explorer</strong> icon) in the left-hand column. It looks like two files on top of each other.</li>



<li>Click the <strong>Open Folder</strong> button.</li>



<li>In the file explorer, select a folder that has some code you want to upload to GitHub, and click <strong>Open</strong>.</li>



<li>After opening your code, select the <strong>Source Control</strong> icon. By default, it&rsquo;s the third icon from the top.</li>



<li>Click the <strong>Initialize Repository</strong> button.</li>
</ol>



<p>At this point, a few things will change in your UI. First, you can see the branch name in the bottom bar on the left-hand side. The default is <code>main</code>. You can rename your branch by using the Command Palette.</p>



<ol class="wp-block-list">
<li>To open the Command Palette, press <code>Shift-Command-P</code> on Mac or <code>Ctrl-Shift-P</code> on PC.</li>



<li>In the Command Palette, start typing &ldquo;rename&rdquo; and select the Git: Rename Branch command.</li>



<li>In the box, provide the new name of the branch and press <code>Enter</code>.</li>
</ol>



<p>At this point, you can see that the name of the branch in the bottom-left corner has been updated to the new name. You can rename it back to <code>main</code> by following the same steps.</p>



<p>Another change you&rsquo;ll see after initializing your repository is that each of the files in the Source Control Panel have a &ldquo;U&rdquo; next to them. &ldquo;U&rdquo; stands for untracked, meaning that these files are new or changed, but have not been added to the repository. To track a file, you just need to click the plus sign adjacent to the file name. If you want to track all of the files, you can click the top plus next to the word <strong>CHANGES</strong>.</p>



<p>When you do this, the file(s) that you select will be staged, and the letter next to them will change to &ldquo;A&rdquo;. This means the file is staged, but not yet uploaded to GitHub. In order to upload the changes, you&rsquo;ll need to commit the files.</p>



<ol class="wp-block-list">
<li>Enter a message in the text box at the top of the Source Control window describing the commit. Alternatively, you could click the <strong>Copilot</strong> icon in the text box to have Copilot generate a commit message for you.</li>



<li>Select the <strong>Commit</strong> button underneath the text box to commit your changes.</li>
</ol>



<p>It&rsquo;s that simple!</p>



<h2 class="wp-block-heading" id="creating-and-changing-branches">Creating and changing branches</h2>



<p>Right now, you&rsquo;re likely on the <code>main</code> branch. Remember that you can check the branch by looking in the bottom-left corner of your window. If this were a major app and you were adding new code or features, you&rsquo;d want to create a new branch and use that for your work.</p>



<ol class="wp-block-list">
<li>Open the Command Palette by pressing <code>Shift-Command-P</code> on Mac or <code>Ctrl-Shift-P</code> on PC.</li>



<li>Enter &ldquo;create branch&rdquo; in the text box.</li>



<li>Select <strong>Git: Create Branch&hellip;</strong> from the list of options.</li>



<li>Enter the name of the new branch in the box. For example, &ldquo;new-features&rdquo;.</li>



<li>Press <code>Enter</code>.</li>
</ol>



<p>Once you do this, VS Code will create the new branch and automatically transfer you to this branch. You can verify this by looking at the branch name in the bottom-left corner.</p>



<h2 class="wp-block-heading" id="tracking-changes-you-make">Tracking changes you make</h2>



<p>Now that you&rsquo;re in your working branch, go ahead and enter a line of code in a file. When you do this, you&rsquo;ll notice that a thin green line appears on the right side of your editor next to the code you added. This section of the editor is called the gutter, and this green line reflects a new line of code that you added.</p>



<p>Move to a different line and make some changes in the line of code that already exists. When you do this, you&rsquo;ll see a blue line with a pattern across it in the gutter. This line indicates that you&rsquo;ve made changes to an existing line of code.</p>



<p>Finally, move to an unchanged line in the file and delete it. Notice that the gutter adds a red arrow. This indicates that a line of code was removed from the file.</p>



<p>When you modify this file, you can see that the file appeared in your Source Control window under the <strong>CHANGES</strong> header. If you hover over the filename, you&rsquo;ll see several buttons appear. There are buttons to open a file, discard your changes, and stage your changes. Similar to before, if you have multiple files with changes, you can hover over the <strong>CHANGES</strong> header and see buttons that will let you review unstaged changes, open the changes, discard all the changes, and stage all the changes.</p>



<h2 class="wp-block-heading" id="viewing-diffs">Viewing diffs</h2>



<p>Sometimes you want to see the changes that you made in a file. VS Code lets you do this by performing side-by-side diffs without needing another tool. To see the changes on a file, click on the name of the file you want to see in the Source Control window. From here you can see the changes in the file and compare the differences.</p>



<p>Depending on your preferences, you can also view your diffs in what is called an inline view.</p>



<ol class="wp-block-list">
<li>Click the three dots (<strong>&hellip;</strong>) in the top-right of the diff view.</li>



<li>Select <strong>Inline View</strong> from the drop-down menu.</li>
</ol>



<p>This lets you see all of the changes in a single window without splitting it up over two separate views. From this view, you can even make edits inside of the diff view.</p>



<p>Once you&rsquo;ve made any changes you want to make to the file, it&rsquo;s time to upload them to GitHub. Following the steps we went over before, go ahead and stage your changes, and then commit your staged changes. Once you finish these steps, you shouldn&rsquo;t have any files displayed in the Source Control window.</p>



<h2 class="wp-block-heading" id="merging-branches">Merging branches</h2>



<p>Note that changes you&rsquo;ve uploaded will still be in your working branch. If you navigate back to the <code>main</code> branch, you&rsquo;ll see the original code before the changes you made.</p>



<ol class="wp-block-list">
<li>Click the branch name in the bottom-left of the window.</li>



<li>Branches names appear in the drop-down at the top of the program. Select the <code>main</code> branch.</li>
</ol>



<p>In order to get these changes into your <code>main</code> branch, you&rsquo;ll need to merge branches.</p>



<ol class="wp-block-list">
<li>In the Source Code window, click the three dots (<strong>&hellip;</strong>) next to <strong>CHANGES</strong>.</li>



<li>In the context menu, hover over <strong>Branch</strong> and select <strong>Merge&hellip;</strong></li>



<li>The box at the top will prompt you with branches that you want to merge from. Select the branch with the changes you want to merge into <code>main</code>.</li>
</ol>



<p>Congratulations! Now your <code>main</code> branch has incorporated the changes!</p>



<h2 class="wp-block-heading" id="publishing-to-github">Publishing to GitHub</h2>



<p>Let&rsquo;s say you want to take your project and publish it up to GitHub. To do so, click the <strong>Publish Branch</strong> button in the Source Control window. VS Code will prompt you with whether you want to publish it as a private or as a public repository. Select the option you want, and then VS Code handles the rest.</p>



<p>Once VS Code finishes the publishing process, it will notify you that the project has been published to a repository on GitHub. You can click the <strong>Open on GitHub</strong> button to visit your project on GitHub and see it online.</p>



<h2 class="wp-block-heading" id="cloning-a-repository">Cloning a repository</h2>



<p>Now let&rsquo;s say you want to clone a repository so that you can work on it on your machine. This creates a local copy that you can use and sync the changes between the two locations. There are multiple ways that you can clone a repository, and this is an easy way to do it in VS Code.</p>



<ol class="wp-block-list">
<li>Navigate to the home page of the repository you want to clone.</li>



<li>Click the green <strong>&lt;&gt; Code</strong> button at the top of the repository file list.</li>



<li>In the drop-down menu, select the <strong>Copy URL to clipboard</strong> button next to the box containing the repository&rsquo;s URL.</li>



<li>Open VS Code.</li>



<li>Open the Command Palette by pressing <code>Shift-Command-P</code> on Mac or <code>Ctrl-Shift-P</code> on PC.</li>



<li>Type in &ldquo;clone&rdquo;.</li>



<li>Select <strong>Git: Clone</strong> from the list of options.</li>



<li>Paste the URL into the box.</li>



<li>Select <strong>Clone from URL</strong> with the URL you pasted after it.</li>



<li>A pop-up window will ask you for a location. Choose the folder where you want to store the project files.</li>



<li>Click the <strong>Select as Repository Destination</strong> button.</li>



<li>A pop-up manu will appear asking if you want to open the repository. Select <strong>Open</strong>.</li>
</ol>



<p>Congratulations! You&rsquo;ve just cloned the repository to your machine and can start to work on it in your local environment!</p>



<h2 class="wp-block-heading" id="model-context-protocol">Model Context Protocol</h2>



<p>Did you know that you don&rsquo;t have to do everything manually in VS Code? You could leverage Model Context Protocol (MCP) to safely let AI tools access external tools and data. The first step is to add the GitHub MCP extension.</p>



<ol class="wp-block-list">
<li>In the left navigation bar, click the <strong>Extensions</strong> icon.</li>



<li>In the search box, enter &ldquo;@mcp github&rdquo;.</li>



<li>Select the GitHub extension from the list.</li>



<li>In the description for the extension, click <strong>Install</strong>.</li>



<li>A pop-up appears, asking you to allow the MCP server to authenticate. Select <strong>Allow.</strong></li>



<li>Select your GitHub username from the list.</li>
</ol>



<p>At this point, the GitHub MCP server is installed. You can verify this by looking at the bottom of the Extensions view and seeing the section for installed MCP servers. <strong>With the MCP server installed, you can use Copilot chat to create some code for you, and it will do so by leveraging external tools where necessary.</strong></p>



<ol class="wp-block-list">
<li>Open the chat window by selecting the <strong>Chat</strong> icon next to the Command Palette window.</li>



<li>Enter a prompt asking Copilot to add some features to your project.</li>
</ol>



<p>For an example of how this works, check out the video version of this episode which walks through asking Copilot to create several issues to improve a flash card application.</p>



<h2 class="wp-block-heading" id="next-steps">Next steps</h2>



<p>And that&rsquo;s it! We covered some of the most common ways developers use VS Code to interact with Git. We&rsquo;ve gone over everything from creating repositories, to publishing on GitHub, and even threw in a little bit of using AI at the end. There are more advanced tips, but these elements are what you&rsquo;ll use most frequently.</p>



<p>Happy coding!</p>

<p>Blockbench is a 3D model editor purpose built for low poly models with pixel art textures. It started life as a Minecraft model editor. That lineage shows in the interface, which is built around cubes, planes, and meshes you can animate without setting up a full rigging pipeline. Since then, the project has grown into a general purpose tool with texture painting, UV mapping, paint directly on the model in 3D, a keyframe animation timeline with a graph editor, a plugin store, and exports to glTF, OBJ, and a long list of game specific formats.</p>



<p><strong>Why it matters</strong>: A focused tool is a fast tool. Because Blockbench commits to one slice of 3D, an artist can go from a blank scene to a textured, animated, exported asset in a single afternoon without learning a full content pipeline first. That low time to first asset is what makes it stick.</p>

<p>Pencil2D is a hand- drawn 2D animation tool aimed at frame-by-frame drawing. It supports bitmap and vector layers, with onion skinning and a redesigned camera system. Perspective grids and adjustable layer and keyframe opacity make clean fades easy. Exports go to image sequences and common video formats, which you can slice into spritesheets for your engine of choice. Pencil2D runs on Windows, macOS, Linux, and FreeBSD, including older hardware most modern creative tools have left behind.</p>



<p><strong>Why it matters</strong>: Pencil2D makes it easy to learn the craft of frame-by-frame animation. Roughing in raster strokes and inking on a vector layer above them happens in one timeline, no round trip to another app. The footprint stays small enough to run on a classroom laptop, which makes it one of the few places a beginner can sit down and learn timing and spacing from first principles.</p>

<p>Pixelorama is a pixel art tool built specifically for game development workflows, which means sprites, tilesets, and animations are first class, not features bolted onto a general purpose paint program. You get onion skinning, tile mode for seamless patterns, an animation timeline, and exports to PNG sequences and spritesheets that drop straight into an engine importer. It is built in Godot and runs on Windows, Linux, macOS, and the web, including a browser build that lets an artist open a project on a Chromebook.</p>



<p><strong>Why it matters</strong>: Because Pixelorama treats sprites and animations as the primary unit of work, the path from idea to a frame in your game is short. Tile mode, onion skinning, and the spritesheet exporter sit one click away on the toolbar instead of buried behind a plugin. That focus is what makes it the natural reach for a 2D pipeline.</p>

<p>Material Maker is a node based procedural texture authoring tool. You build a graph of generators, filters, and blends. Out the other end, you get PBR texture sets ready for a real time renderer. Like Pixelorama, it is built in Godot, which makes it a second entry on this list where an open source content tool runs on top of an open source game engine.</p>



<p><strong>Why it matters</strong>: Procedural authoring trades a one-off painted texture for a graph you can constantly adjust for new textures. Crank up the moss and instead of a newly painted stone wall you have one that looks like it&rsquo;s been sitting in the woods for a decade. Multiply that across every surface in your game, and texture work starts to scale with the project instead of fighting it every time the art direction shifts.</p>

<p>LDtk is an entity-driven 2D level editor, which means you define your entity types up front, set up auto tiling rules, and use enums to keep your data clean as the project grows. 					<description><![CDATA[<p>Explore our update on GitHub’s accessibility strategy, and learn how you can join us in building a culture of accessibility.</p>
<p>The post <a href="https://github.blog/open-source/building-githubs-next-chapter-in-accessibility/">Building GitHub&#8217;s next chapter in accessibility</a> appeared first on <a href="https://github.blog">The GitHub Blog</a>.</p>

										<content:encoded><![CDATA[<!DOCTYPE html PUBLIC "-//W3C//DTD HTML 4.0 Transitional//EN" "http://www.w3.org/TR/REC-html40/loose.dtd">
<html><body><p>Five years ago, we stood up GitHub&rsquo;s accessibility program. What began as a small team addressing accessibility debt has grown into a company-wide discipline, woven into our engineering fundamentals, our design system, our AI tools, and our culture. The five-year milestone prompted us to step back and ask a fundamental question: where do we go from here?</p>



<p>The answer is our <a href="https://accessibility.github.com/">strategy for accessibility</a>, which we published earlier this year. The strategy marks a pivotal moment for our program. For the first five years, our focus was primarily internal. Going forward, we are turning outward&mdash;engaging the global community of developers as we continue to mature internally. We want to build a culture of accessibility across GitHub and include every developer, every team, and every open source project.</p>



<p>The strategy is organized around four priorities. On this Global Accessibility Awareness Day, we&rsquo;re sharing initial progress on each strategic priority.</p>



<h2 class="wp-block-heading" id="h-help-improve-the-accessibility-of-open-source-at-scale">Help improve the accessibility of open source at scale</h2>



<p>Open source software powers much of the world&rsquo;s technology, yet many projects are not designed to be usable by people with disabilities. On the eve of GAAD 2025, we took a <a href="https://github.blog/open-source/social-impact/our-pledge-to-help-improve-the-accessibility-of-open-source-software-at-scale/">pledge to help change that</a>. We committed to three goals: empower people with disabilities to contribute to open source, increase the availability of open source assistive technologies, and improve the accessibility of mainstream open source projects.</p>



<p>Last year, we launched an open source initiative that is turning our pledge into action.</p>



<h3 class="wp-block-heading" id="h-open-source-assistive-technology-hackathon">Open Source Assistive Technology Hackathon</h3>



<p>This week, we&rsquo;re hosting the first <a href="https://github.com/open-source-accessibility/Open-Source-Assistive-Technology-Hackathon">Open Source Assistive Technology Hackathon</a> at GitHub headquarters in San Francisco. Over two days, participants will contribute to 16 featured projects that empower people with disabilities. They include projects that enable blind students to interact with graphical information on the Monarch refreshable tactile display, a project that leverages AI to convert PDF files into an accessible format, hacks for power wheelchairs, and much more. The hackathon also features Office Hours for the open source NVDA screen reader and a GitHub Learning Room to help participants learn open source workflows on GitHub.</p>



<h3 class="wp-block-heading" id="h-open-source-accessibility-summit">Open Source Accessibility Summit</h3>



<p>In October 2025, we hosted the inaugural <a href="https://2025.allthingsopen.org/open-source-accessibility-summit">Open Source Accessibility Summit</a> at All Things Open in Raleigh, North Carolina. The response was overwhelming&mdash;300 people registered and more than 500 joined the waitlist. The summit brought together experts from the disability, accessibility, and open source communities to identify six priority challenge areas and draft a collaborative roadmap. That work continues in the <a href="https://github.com/open-source-accessibility">open-source-accessibility</a> organization on GitHub, with community discussions coordinated through a public Slack workspace.</p>



<h3 class="wp-block-heading" id="h-accessibility-best-practices-for-open-source-projects">Accessibility best practices for open source projects</h3>



<p>Maria Lamardo collaborated with open source maintainers to publish <a href="https://opensource.guide/accessibility-best-practices-for-your-project/">accessibility best practices for open source projects</a> on opensource.guide. The guide helps maintainers center people with disabilities in their development process&mdash;from writing an accessibility statement and making documentation accessible by default, to designing keyboard-navigable interfaces and using semantic HTML.</p>



<h2 class="wp-block-heading" id="h-empower-developers-with-disabilities-to-build-on-github">Empower developers with disabilities to build on GitHub</h2>



<p>As the home for all developers, we want every developer to feel welcome and to be able to contribute to the future of global software development using everything GitHub has to offer. Accessibility is a <a href="https://github.blog/engineering/engineering-principles/githubs-engineering-fundamentals-program-how-we-deliver-on-availability-security-and-accessibility/">GitHub Engineering Fundamental</a>. We drive excellence by clearly defining expectations, continuously testing against them, and ensuring accountability through our engineering scorecards. Our <a href="https://primer.style/">Primer Design System</a> provides a strong foundation, and we embed accessibility designers within product teams to ensure accessibility is considered from the first sketch to the final deployment.</p>



<p>Over the past year, we&rsquo;ve made substantial improvements across the platform.</p>



<h3 class="wp-block-heading" id="h-a-redesigned-pull-request-experience">A redesigned pull request experience</h3>



<p>Pull requests are an essential part of the developer experience. This year, the pull requests team completely redesigned the files changed page and included accessibility at every step in the process. They rebuilt it with consistent keyboard navigation, landmarks, adjustable line spacing, and fewer page reloads (critical for screen reader users who lose their place when pages refresh). They shipped seven updates over seven months, and the improved experience became the <a href="https://github.blog/changelog/2026-01-22-improved-pull-request-files-changed-page-on-by-default/">default for all users</a> in January 2026.</p>



<h3 class="wp-block-heading" id="h-enhanced-contrast-and-themes">Enhanced contrast and themes</h3>



<p>In June 2025, the Design team <a href="https://github.blog/changelog/2025-06-05-control-contrast-for-all-github-themes/">introduced enhanced contrast controls</a> for all GitHub themes. For the first time, contrast is adjustable for logged-out users&mdash;anyone who visits GitHub can customize their visual experience without needing an account.</p>



<h3 class="wp-block-heading" id="h-smarter-search">Smarter search</h3>



<p>In April 2026, <a href="https://github.blog/changelog/2026-04-02-improved-search-for-github-issues-is-now-generally-available/">semantic search for GitHub Issues</a> reached general availability. Users can now describe what they&rsquo;re looking for in natural language and find conceptually related results, reducing cognitive load and making the platform more intuitive for everyone.</p>



<h3 class="wp-block-heading" id="h-major-accessibility-improvements-for-github-command-line-interfaces-clis">Major accessibility improvements for GitHub Command Line Interfaces (CLIs)</h3>



<p>We believe the terminal is essential to the developer experience, and it has been an underserved environment for accessibility across the industry. Over the past year, our CLI team raised the bar.</p>



<p>In May 2025, the CLI team launched <a href="https://github.blog/engineering/user-experience/building-a-more-accessible-github-cli/">accessibility improvements to the GitHub CLI</a>, introducing screen reader support that replaced prompts and spinner indicators that confused speech synthesis with accessible alternatives. They added customizable color palettes aligned to ANSI 4-bit colors, giving users with low vision and colorblindness full control over their terminal experience.</p>



<p>They carried these principles forward into the development of <a href="https://github.blog/changelog/2026-02-25-github-copilot-cli-is-now-generally-available/">GitHub Copilot CLI</a>, which reached general availability in February 2026. Copilot CLI brings the power of GitHub Copilot coding agent directly to the terminal&mdash;and it shipped with accessibility built in from day one:</p>



<ul class="wp-block-list">
<li><strong>Dedicated screen reader mode</strong> (<code>--screen-reader</code>) with consistent dialog titles, reduced redundant announcements, and enhanced contrast.</li>



<li><strong>Theme picker</strong> with colorblind-friendly variants including GitHub Dark, GitHub Light, and high-contrast options.</li>



<li><strong>Full keyboard-first navigation</strong> with UNIX keybindings, quick help overlay, and configurable reasoning visibility.</li>



<li><strong>Responsive layout </strong>that adapts to narrow terminals and different screen configurations.</li>
</ul>



<p>Finally, we published a <a href="https://accessibility.github.com/documentation/guide/cli/" target="_blank" rel="noreferrer noopener">guide for using Git, GitHub CLI, and GitHub Copilot CLI with a screen reader</a>. It includes step-by-step installer walkthroughs with screen reader&ndash;specific notes and end-to-end workflows that chain commands together, so blind developers can go from zero to productive on the command line as quickly as possible.</p>



<h2 class="wp-block-heading" id="h-empower-our-customers-to-meet-their-accessibility-goals-on-github">Empower our customers to meet their accessibility goals on GitHub</h2>



<p>We are eager to help our customers meet their accessibility goals when they build on GitHub. Our approach is to lead by example, using the latest GitHub features to run our own accessibility program, sharing our process improvements transparently, and open-sourcing the tools we build.</p>



<h3 class="wp-block-heading" id="h-sharing-what-we-ve-learned">Sharing what we&rsquo;ve learned</h3>



<p>This year, we published detailed accounts of how we run our accessibility program, so our customers can learn from our experience. Janice Rimmer&rsquo;s post on <a href="https://github.blog/ai-and-ml/github-copilot/how-we-automated-accessibility-compliance-in-five-hours-with-github-copilot/" target="_blank" rel="noreferrer noopener">automating accessibility governance with Copilot</a> showed how a program manager&mdash;not an engineer&mdash;can build automation that transforms compliance workflows. Carie Fisher&rsquo;s post on <a href="https://github.blog/ai-and-ml/github-copilot/continuous-ai-for-accessibility-how-github-transforms-feedback-into-inclusion/">Continuous AI for Accessibility</a> detailed our user feedback pipeline, where GitHub Copilot analyzes incoming reports and auto-populates approximately 80% of issue metadata. That workflow has driven a <strong>62% reduction in resolution time</strong> and ensured <strong>89% of issues close within 90 days</strong>. Finally, Eric Bailey shared timely insights about accessibility agents in his post about <a href="https://github.blog/ai-and-ml/github-copilot/building-a-general-purpose-accessibility-agent-and-what-we-learned-in-the-process/">Building our general-purpose accessibility agent</a>.</p>



<h3 class="wp-block-heading" id="h-figma-annotation-toolkit">Figma Annotation Toolkit</h3>



<p>When we analyzed our accessibility audit data, we found that <strong>48% of issues could have been prevented in the design phase</strong>. That insight led Jan Maarten and our Accessibility Design team to build the <a href="https://github.com/github/annotation-toolkit">Annotation Toolkit</a>, a comprehensive Figma library that helps designers document accessibility intent directly in their work, from heading hierarchies and keyboard navigation flows to ARIA semantics and screen reader announcements. They <a href="https://github.blog/enterprise-software/collaboration/level-up-design-to-code-collaboration-with-githubs-open-source-annotation-toolkit/">open-sourced the toolkit</a> in September 2025 so any team can use it.</p>



<h3 class="wp-block-heading" id="h-ai-powered-accessibility-scanner">AI-powered accessibility scanner</h3>



<p>Our Accessibility Engineering and State and Local Government revenue teams collaborated to build an <a href="https://github.com/marketplace/actions/accessibility-scanner">AI-powered accessibility scanner</a> that enables customers to find, file, and fix accessibility bugs using GitHub Copilot cloud agent. The scanner uses the highly-trusted open source axe-core library from Deque Systems to find accessibility barriers through static DOM analysis. Most recently, the team implemented a new plugin architecture and an initial built-in plugin that detects <a href="https://www.w3.org/WAI/WCAG21/Understanding/reflow">WCAG 1.4.10 Reflow violations</a>. The scanner is available in the <a href="https://github.com/marketplace/actions/accessibility-scanner">GitHub Marketplace</a> and as an <a href="https://github.com/github/accessibility-scanner">open-source repository</a> that teams can fork and adapt to their unique CI/CD processes.</p>



<h3 class="wp-block-heading" id="h-accessibility-guides-for-developers">Accessibility guides for developers</h3>



<p>Kendall Gassner published a <a href="https://accessibility.github.com/documentation/guide/copilot-instructions/" target="_blank" rel="noreferrer noopener">guide for optimizing GitHub Copilot for accessibility using custom instructions</a>, helping developers optimize GitHub Copilot for accessibility for their teams. Roberto Perez published a <a href="https://accessibility.github.com/documentation/guide/getting-started-with-agents/" target="_blank" rel="noreferrer noopener">guide for creating custom agents for accessibility</a> that can jumpstart experimentation with custom agents for accessibility.</p>



<h3 class="wp-block-heading" id="h-the-github-enterprise-accessibility-advisory-panel-gaap">The GitHub Enterprise Accessibility Advisory Panel (GAAP)</h3>



<p>In April 2026, we launched the <a href="https://accessibility.github.com/accessibility-advisory-panel.html">GitHub Enterprise Accessibility Advisory Panel (GAAP)</a>, a forum for regular exchange between GitHub and enterprise customers working to build accessible software on GitHub. GAAP bridges real-world accessibility challenges and GitHub&rsquo;s platform features, with an emphasis on adoption of current capabilities and identification of future needs. Membership is open to accessibility professionals from GitHub enterprise customer organizations.</p>



<h2 class="wp-block-heading" id="h-empower-hubbers-with-disabilities-to-do-the-best-work-of-their-lives">Empower Hubbers with disabilities to do the best work of their lives</h2>



<p>We know that <a href="https://www.accenture.com/content/dam/accenture/final/a-com-migration/pdf/pdf-89/accenture-disability-inclusion-research-report.pdf" target="_blank" rel="noreferrer noopener">companies that embrace best practices for employing and supporting people with disabilities outperform their peers</a>, and we take that responsibility seriously.</p>



<p>All employees (we call them Hubbers) are required to complete accessibility training. We integrate accessibility into our procurement processes to continuously improve the tools and systems our employees depend on. We provide guides that help every employee create accessible communications, events, and meetings. And our NeuroCats Community of Belonging and AccessCats Affinity Group give employees with disabilities a strong voice within the company.</p>



<p>Over the past year, our People team updated the categories that Hubbers use to self-identify, which allows us to better represent the changing demographics within GitHub. While the AccessCats team completed a survey of disability at GitHub that will enable us to better serve Hubbers with disabilities.</p>



<h2 class="wp-block-heading" id="h-join-us">Join us</h2>



<p>Accessibility is never done. Publishing our <a href="https://accessibility.github.com/">strategy</a> is not the finish line&mdash;it is the starting line for the next chapter. We are building in public, sharing our tools, and inviting the global developer community to join us.</p>

<li><strong>Try our tools</strong>: the <a href="https://github.com/github/annotation-toolkit">Annotation Toolkit</a> and the <a href="https://github.com/marketplace/actions/accessibility-scanner">accessibility scanner</a>.</li>



<li><strong>Join the open source accessibility movement</strong> at <a href="https://github.com/open-source-accessibility">github.com/open-source-accessibility</a>.</li>

<p>Given the reality of threat actors and the advent of AI technologies, we need to do all we can to protect our customers. <li><a href="https://www.w3.org/WAI/WCAG22/Understanding/non-text-content.html">Ensuring there are text alternatives for non-text content</a></li>



<li><a href="https://www.w3.org/WAI/WCAG22/Understanding/focus-order.html">Moving keyboard focus through pages and views in a logical order</a></li>

<h3 class="wp-block-heading h5-mktg gh-aside-title is-typography-preset-h5" id="h-what-are-llms-and-agents" style="margin-top:0">What are LLMs and agents?</h3>



<p><strong>Note</strong>: This post assumes at least some understanding of LLMs, agents, and their related concepts. If you&rsquo;re not familiar, you can <a href="https://github.blog/ai-and-ml/llms/">learn more about LLMs on our blog</a>.</p>

</ul>

<h2 class="wp-block-heading" id="h-mindset">Mindset</h2>



<p><a href="https://en.wikipedia.org/wiki/Social_model_of_disability">The social model of disability</a> teaches us that access barriers&mdash;and consequently impairment&mdash;can be created because of how an environment is built. The same thinking applies to digital experiences.</p>



<p>With the accessibility agent, <strong>we are not attempting to &ldquo;solve&rdquo; accessibility</strong> in isolation. We are instead attempting to <strong>augment our peers&rsquo; efforts</strong>, to better help them remove the barriers that may be created as a result of how we construct GitHub&rsquo;s user interfaces.</p>



<p><strong>The accessibility agent is not a &ldquo;silver bullet&rdquo;</strong> that can automatically address every hypothetical scenario. Understanding, honoring, and socializing this better helps set the agent&rsquo;s scope of responsibility. This sped up the experiment&rsquo;s launch, leading to more buy-in for the effort.</p>



<h2 class="wp-block-heading" id="past-efforts">Past efforts</h2>



<p><a href="https://commission.europa.eu/strategy-and-policy/policies/justice-and-fundamental-rights/disability/european-accessibility-act-eaa_en">The European Accessibility Act</a> is now in effect. <a href="https://www.ada.gov/law-and-regs/regulations/title-ii-2010-regulations/">Title II of the Americans with Disabilities Act</a> is set to establish meeting WCAG 2.1 AA as the legal definition of done in April of 2027. LLM agents can read and take action on <a href="https://www.nngroup.com/articles/ai-agents-as-users/#:~:text=Accessibility%2DTree%20Parsing">the accessibility tree</a>.</p>



<p>To say it plainly: <strong>Organizations will be at a disadvantage if they have not already invested in manually identifying and remediating accessibility issues</strong>. There are many reasons for this, including building an accessibility agent.</p>



<p>To that point, GitHub has a mature system in place for logging accessibility issues, as well as verifying fixes to issues are working as intended. This includes:</p>



<ul class="wp-block-list">
<li>A structured template for reporting problems</li>



<li>Steps to reproduce the issue</li>



<li>A rich layer of metadata about the issue&rsquo;s severity level, service area, and applicable <a href="https://www.w3.org/TR/WCAG22/">WCAG success criterion</a></li>



<li>Crosslinks to the Pull Request that addressed the issue</li>



<li>Acceptance criteria</li>
</ul>



<p>In addition, all the issues are centralized to a single repository. While this issue-logging effort predated the explosion in popularity of LLM tooling, its highly consistent and structured nature made it <strong>an ideal corpus of content</strong> for the accessibility agent to reference.</p>



<p>Because of this, we instructed the agent to investigate these issues to see if there are related code and language snippets it can extrapolate from. This is one area where the non-deterministic &ldquo;fuzzy matching&rdquo; behavior of LLMs acts as an asset rather than a possible liability.</p>



<h3 class="wp-block-heading" id="old-gold">Old gold</h3>



<p>Much like with any other specialized domain area, vague instructions in a skill file won&rsquo;t cut it. Telling an LLM to &ldquo;use accessibility best practices&rdquo; with a short list of examples won&rsquo;t work well.</p>



<p>When generating code, <a href="https://microsoft.github.io/a11y-llm-eval-report/"><strong>LLMs have an unfortunate bias towards producing accessibility antipatterns</strong></a> since every major LLM currently available is trained on decades of inaccessible code.</p>



<p>To counteract this, the agent needs better content to draw from.</p>



<p>So, <strong>I enthusiastically recommend investing in manually cataloging and remediating accessibility issues</strong>. After some progress, this data can be incorporated into the agent.</p>



<p>The issues and their corresponding pull requests provide highly contextual examples for the LMM to reference, written using the conventions set up by the organization it is deployed in. This collection of issues and code is by far one of the strongest assets the agent draws from.</p>



<h2 class="wp-block-heading" id="efficient-token-consumption">Efficient token consumption</h2>



<p><strong>Accessibility is a holistic concern</strong>, intersecting with code, design, copywriting, and numerous other disciplines involved with creating user interfaces.</p>



<p>A lot of accessibility work is also <strong>highly contextual</strong>, meaning that someone typically needs the full working picture of a problem before they&rsquo;re able to give the appropriate advice for what to do.</p>



<p>Because of these two factors, a general-purpose accessibility agent can consume a ton of tokens when it performs work. This has three negative outcomes:</p>



<ol class="wp-block-list">
<li>An increased amount of unreliable output</li>



<li>Slower response times</li>



<li>Increased operational costs</li>
</ol>



<p>It&rsquo;s important to be diligent when structuring the agent. Here&rsquo;s how we went about doing just that.</p>



<h3 class="wp-block-heading" id="h-use-sub-agents">Use sub-agents</h3>



<p>The accessibility agent started as a single monolithic agent, but quickly grew past the limitations of this approach. Because of this, we evolved it to use a <a href="https://dev.to/goose_oss/agents-subagents-and-multi-agents-what-they-are-and-when-to-use-them-39na#:~:text=The%20Subagent%20Setup%3A%20Orchestrator%20With%20a%20Crew">sub-agent architecture</a>.</p>



<p>A lot of guides recommend creating a large suite of sub-agents, each with its own specific area of responsibility. Here, the sub-agents are executed in parallel, with the main agent reconciling their output.</p>



<p>Surprisingly, <strong>this approach worked against us</strong> for the accessibility agent. Instead, we wound up using two dedicated sub-agents:</p>



<ol class="wp-block-list">
<li>The first sub-agent acts as a <strong>passive reviewer and researcher.</strong></li>



<li>The second sub-agent acts as an <strong>active implementer.</strong></li>
</ol>



<p>The two sub-agents are sandboxed and cannot directly pass content to each other. Instead, they generate a structured, templatized output. This output is then served to the parent orchestrating accessibility agent to consume, validate, and route.</p>



<figure class="wp-block-image size-large"><img data-recalc-dims="1" loading="lazy" decoding="async" height="731" width="1024" src="https://github.blog/wp-content/uploads/2026/05/image2.png?resize=1024%2C731" alt="A diagram demonstrating how the parent accessibility agent passes work sequentially from itself to a read-only reviewer sub-agent, then back to the parent agent, to a write and read-capable implementer sub-agent, then back again to the parent agent. The parent agent is contained in a column labeled, 'Tier 1 - Orchestration', and the two sub-agents are contained in a column labeled, 'Tier 2 - Specialists'. The first connecting line that shows the parent agent passing work off to the reviewer sub-agent is labeled, 'run sub-agent'. The second line that passes work back to the parent agent is labeled, 'structured findings'. The third line has the parent agent passing work to the implementer sub-agent, and is labeled 'Run sub-agent with structured findings'. The fourth and final line passes work from the implementer sub-agent back to the parent agent and is labeled, 'Changes or guidance generated'. The parent and sub-agents also have lists of responsibilities. The parent accessibility agent routes requests, locates code and skills, runs complexity scoring, validates outputs, manages escalation gates, manages re-audit loops, and answers research questions. The reviewer sub-agent  performs code audits, WCAG research, detects escalation triggers, and produces structured findings. The implementer sub-agent has two modes: a default code-change mode and a fallback guidance-only mode. The code-change mode fixes critical issues first, then addresses the rest. The guidance-only mode generates guidance docs. Both modes validates changes." class="wp-image-96094" srcset="https://github.blog/wp-content/uploads/2026/05/image2.png?w=3000 3000w, https://github.blog/wp-content/uploads/2026/05/image2.png?w=300 300w, https://github.blog/wp-content/uploads/2026/05/image2.png?w=768 768w, https://github.blog/wp-content/uploads/2026/05/image2.png?w=1024 1024w, https://github.blog/wp-content/uploads/2026/05/image2.png?w=1536 1536w, https://github.blog/wp-content/uploads/2026/05/image2.png?w=2048 2048w" sizes="auto, (max-width: 1000px) 100vw, 1000px" /></figure>



<p>There are a few reasons for this approach:</p>



<ul class="wp-block-list">
<li><strong>Escalation checkpoints</strong>. The reviewer checks for areas where human intervention will likely be needed. This includes multiple high-severity WCAG failures, as well as a list of patterns that are known to be difficult to make accessible.</li>



<li><strong>Complexity-based behavior</strong>. The agent is instructed to operate in a specialized guidance-only mode if the underlying code is deemed too complicated. Here, the parent accessibility agent acts as an arbiter, while the reviewer agent is &ldquo;opinionless&rdquo; and just reports the findings as instructed.</li>



<li><strong>Filtering</strong>. The reviewer presents everything it finds. The parent accessibility agent then utilizes resources and skills to determine what is relevant to the request. The reviewer passing all its findings to the implementer would be costly and potentially set it on irrelevant and counter-productive tasks.</li>



<li><strong>Traceability</strong>. Direct communication between sub-agents would remove the ability to create and review an audit trail of user and agent decisions. This is important given the agent&rsquo;s instruction around complex patterns, as well as the highly contextual nature of accessibility work.</li>
</ul>



<h3 class="wp-block-heading" id="execute-instructions-in-a-linear-order">Execute instructions in a linear order</h3>



<p>In addition to being a holistic concern, effective digital accessibility work also demands a methodical, detail-oriented approach.</p>



<p>The concern of using sub-agents to increase the speed of the LLM&rsquo;s reply is counterbalanced by our need for its results to be accurate. We found that <strong>compelling the agent to execute its sub-agent instructions in a fixed order</strong> was key.</p>



<p>We first establish a parent set of ordered phases. Each phase itself contains child ordered steps of instructions, which are accompanied by relevant resources and skills:</p>



<figure class="wp-block-image size-large"><img data-recalc-dims="1" loading="lazy" decoding="async" height="592" width="1024" src="https://github.blog/wp-content/uploads/2026/05/image3.png?resize=1024%2C592" alt="A diagram demonstrating how the research sub-agent uses ordered phases and ordered steps within each phase to produce structured output. <p>The interesting bit about this linear order is that it mirrors how I would personally approach performing auditing, remediating, and reporting duties.</p>



<h3 class="wp-block-heading" id="use-a-template-schema-pass-around-sub-agent-content">Use a template schema pass around sub-agent content</h3>



<p>The entire operation of the sandboxed sub-agents is built around template schema files. These files <strong>create consistency</strong> that is vital to keeping the agent focused and on track.</p>



<p>The two schema templates are:</p>



<ol class="wp-block-list">
<li><strong>Reviewer template schema:</strong> This focuses on what to audit, and how to find applicable information about it.</li>



<li><strong>Implementer template schema:</strong> This focuses on what to fix and how to fix it.</li>
</ol>



<p>Without the schema files in place, the agents would all attempt to arbitrarily communicate with each other. This would create increased token expenditure, undesirable hallucinations, unnecessary code changes, and difficult-to-impossible behavior for agent auditing purposes.</p>



<h2 class="wp-block-heading" id="acknowledging-limitations">Acknowledging limitations</h2>



<p>Another vital aspect of creating the accessibility agent is <strong>understanding areas where agents can fall short</strong>.</p>



<p>As the agent is not a turnkey &ldquo;solution&rdquo; for accessibility, we want to avoid situations where the agent&rsquo;s output in error may not be sufficiently interrogated by the human using it. This is especially relevant when someone is not well-versed in digital accessibility considerations and practices.</p>



<p>Here&rsquo;s what we did to accommodate the agent&rsquo;s limitations:</p>



<h3 class="wp-block-heading" id="evaluate-code-complexity">Evaluate code complexity</h3>



<p>We want to avoid scenarios where we would need to perform costly and time-intensive work to revisit an inaccessible solution that the agent &ldquo;thinks&rdquo; is accessible.</p>



<p>To aid with this problem, the accessibility agent uses a small shell script to analyze the code it is set to work on. The script itself is simple, using a small set of basic heuristics to evaluate the relative complexity and distill it down into a score.</p>



<p>This score is then ingested by the agent. If the score passes a set threshold, the agent is instructed to <strong>not</strong> execute code changes. Instead, it will inform the person using the LLM that they should reach out to the accessibility team to consult on what they are attempting to do.</p>



<h3 class="wp-block-heading" id="identify-high-risk-patterns">Identify high-risk patterns</h3>



<p>It is a subtle thing to understand, but know that <strong>it is entirely possible to have code that passes automated accessibility checks, yet is functionally unusable</strong>.</p>



<p>As a companion to code complexity, the accessibility agent is instructed to avoid attempting code generation for patterns the accessibility team has identified as high-risk. This includes, but is not limited to: drag and drop, <a href="https://primer.style/accessibility/patterns/accessible-notifications-and-messages/#toasts">toasts</a>, rich text editors, <a href="https://github.blog/engineering/user-experience/considerations-for-making-a-tree-view-component-accessible/">tree views</a>, and <a href="https://www.w3.org/WAI/ARIA/apg/patterns/grid/">data grids</a>.</p>



<p>These patterns require a ton of focused attention and detail and currently sit outside of an LLM&rsquo;s current capabilities to produce in a way that actually works with assistive technology.</p>



<p>Not prohibiting high-risk patterns and high-complexity code environments would lead to unnecessary demands of everyone&rsquo;s time to readdress, and also represents reputational risk for the accessibility team. We avoid this by shutting off the LLMs capability to go down this pathway.</p>



<h3 class="wp-block-heading" id="reduce-bias-to-action">Reduce bias to action</h3>



<p>I am loathe to anthropomorphize LLMs, but one quality they all seem to share is <strong>desperately wanting to produce content</strong>. For Copilot, that often means generating code.</p>



<p>We had to create <a href="https://en.wiktionary.org/wiki/game_the_system">anti-gaming</a> instructions to prevent the LLM from creating sneaky ways to get around its instructions to not generate code when human expertise is needed. This prevented it from violating its own intervention instructions.</p>



<h3 class="wp-block-heading" id="know-that-programmatically-determinable-issues-dont-cover-everything">Know that programmatically determinable issues don&rsquo;t cover everything</h3>



<p>Agent success metrics live within a larger context.</p>



<p>Of the 55 total WCAG level A and AA Success Criterion, only <a href="https://www.w3.org/WAI/standards-guidelines/act/rules/?requirements=a%2Caa&amp;status=approved&amp;implement=auto&amp;display=no-rules">35 of them can be detected via deterministic automated code checkers</a>. This means that <strong>~36% of level A and AA Success Criterion cannot be discovered automatically</strong>.</p>



<figure class="wp-block-image size-full"><img data-recalc-dims="1" loading="lazy" decoding="async" width="729" height="807" src="https://github.blog/wp-content/uploads/2026/05/image4.png?resize=729%2C807" alt="A pie chart titled, 'WCAG A and AA Success Criterion'. The first of two slices is labeled, '36% require manual evaluation'. The second of two slices is labeled, '64% can be detected automatically'." class="wp-image-96096" srcset="https://github.blog/wp-content/uploads/2026/05/image4.png?w=729 729w, https://github.blog/wp-content/uploads/2026/05/image4.png?w=271 271w" sizes="auto, (max-width: 729px) 100vw, 729px" /></figure>



<p>LLM-powered agent operation is making inroads on this ~36% gap, but it is not a perfect science. Because of this, it becomes important to <strong>manually identify accessibility barriers earlier during design and prototyping efforts</strong>&mdash;<a href="https://www.deque.com/blog/auditing-design-systems-for-accessibility/#design-should-consider-accessibility">the area where the majority of accessibility issues originate</a>.</p>



<p>This thinking is also reflected in the agent&rsquo;s escalation logic, in that members of the Accessibility team can pair with designers to help consider alternate approaches and brainstorm solutions that achieve business goals without compromising on accessibility.</p>



<p>This intervention and assistance is done to thwart potential downstream issues&mdash;and costly and time-consuming redesigns&mdash;are stopped before they ever have a chance to get off the ground.</p>



<h3 class="wp-block-heading" id="manually-evaluate-agent-output-and-adjust-things-that-arent-working-as-expected">Manually evaluate agent output and adjust things that aren&rsquo;t working as expected</h3>



<p>We periodically perform manual review of agent output to determine its accuracy and efficacy. In addition, we have tooling in place to capture pull request reviewer sentiment. Both serve as <strong>strong signals for areas where the agent needs better instruction</strong>, as well as new resources and skills.</p>



<h2 class="wp-block-heading" id="learning-in-the-open">Learning in the open</h2>



<p>To recap, we learned that the agent is:</p>



<ul class="wp-block-list">
<li>Used to aid and augment existing accessibility efforts, not to replace them.</li>



<li>Significantly more effective when trained on manually audited and remediated accessibility issues for your specific experience.</li>



<li>Far more efficient with token consumption when utilizing sub-agents.</li>



<li>More accurate and effective when executing instructions in a methodical, linear fashion.</li>



<li>More consistent when set to use preformatted templates to pass information around.</li>



<li>Set to understand its limitations and route people to alternative support systems.</li>



<li>Improved when its output is periodically reviewed to identify areas it needs better instruction.</li>
</ul>



<p>This journey is also not yet complete. The accessibility agent continues to be iterated upon in the hopes of <strong>helping ensure GitHub is an accessible and inclusive platform for all developers</strong>.</p>



<p>We hope that we can eventually open source the agent as part of <a href="https://github.blog/open-source/social-impact/our-pledge-to-help-improve-the-accessibility-of-open-source-software-at-scale/">our pledge to help improve the accessibility of open source software at scale</a>. Until then, we hope that in sharing our learnings with this undertaking that other teams can have a resource to reference for their own accessibility efforts.</p>

		<title>Raising the bar: Quality, shared responsibility, and the future of GitHub&#8217;s bug bounty program</title>
		<link>https://github.blog/security/raising-the-bar-quality-shared-responsibility-and-the-future-of-githubs-bug-bounty-program/</link>

					<description><![CDATA[<p>We're updating our bug bounty program standards to prioritize quality submissions, clarify shared responsibility boundaries, and evolve how we reward low-risk findings.</p>
<p>The post <a href="https://github.blog/security/raising-the-bar-quality-shared-responsibility-and-the-future-of-githubs-bug-bounty-program/">Raising the bar: Quality, shared responsibility, and the future of GitHub&#8217;s bug bounty program</a> appeared first on <a href="https://github.blog">The GitHub Blog</a>.</p>

										<content:encoded><![CDATA[<!DOCTYPE html PUBLIC "-//W3C//DTD HTML 4.0 Transitional//EN" "http://www.w3.org/TR/REC-html40/loose.dtd">
<html><body><p>The security research community is one of GitHub&rsquo;s greatest assets. Every year, researchers from around the world help us find and fix vulnerabilities, making the platform safer for over 180 million developers. Our bug bounty program exists because we believe that collaboration with external researchers is one of the most effective ways to improve security, and we remain deeply committed to it.</p>



<p>But like every bug bounty program, we&rsquo;re adapting to a changing landscape. We want to share what we&rsquo;re seeing, what we&rsquo;re doing about it, and how we think about the security boundaries of a platform like GitHub.</p>



<h2 class="wp-block-heading" id="h-the-volume-problem">The volume problem</h2>



<p>Over the past year, submission volume across the industry has grown significantly. New tools, including AI, have lowered the barrier to entry for security research, which in many ways is a positive development. More people exploring attack surfaces means more opportunities to find real issues.</p>



<p>However, alongside the growth in legitimate reports, we&rsquo;ve seen a sharp increase in submissions that don&rsquo;t demonstrate real security impact. These include reports without a proof of concept, theoretical attack scenarios that don&rsquo;t hold up under scrutiny, and findings that are already covered by our published ineligible list. This isn&rsquo;t unique to GitHub. Programs across the industry are grappling with the same challenge, and some have shut down entirely.</p>



<p>We don&rsquo;t want to go that direction. Instead, we want to invest in making our program better.</p>



<h2 class="wp-block-heading" id="h-what-makes-a-strong-submission">What makes a strong submission</h2>



<p>We&rsquo;re raising the bar on what we consider a complete submission. Going forward, reports will be evaluated more strictly against these criteria:</p>



<ul class="wp-block-list">
<li><strong>A working proof of concept with demonstrated security impact.</strong> Show us the impact, don&rsquo;t just describe it. What could an attacker actually achieve? We need a working proof of concept that demonstrates real exploitation and concrete security impact. Show us the boundary that can be crossed, not just that one theoretically exists. If your report says &ldquo;this could lead to&hellip;&rdquo; but doesn&rsquo;t show that it <em>does</em>, it&rsquo;s incomplete.</li>



<li><strong>Awareness of scope and ineligible findings.</strong> Before submitting, review our <a href="https://bounty.github.com/scope" type="link" id="https://bounty.github.com/scope" target="_blank" rel="noreferrer noopener">scope</a> and <a href="https://bounty.github.com/ineligible" target="_blank" rel="noreferrer noopener">ineligible findings list</a>. Reports covering known ineligible categories (DMARC/SPF/DKIM configuration, user enumeration, missing security headers without a demonstrated attack path, and others) will be closed as Not Applicable, which may impact your HackerOne Signal and reputation.</li>



<li><strong>Validation before submission.</strong> No matter what tools you use (scanners, static analysis, AI assistants), you need to validate the output before submitting. A false positive that&rsquo;s been manually reviewed is caught before it wastes anyone&rsquo;s time. One that hasn&rsquo;t is just noise.</li>
</ul>



<h2 class="wp-block-heading" id="h-we-welcome-ai-in-security-research">We welcome AI in security research</h2>



<p>We want to be explicit about this: <strong>we have no problem with researchers using AI tools.</strong> AI is a force multiplier, and we expect it to play an increasing role in security research. We use AI across our own internal security programs, and we&rsquo;re seeing the best external researchers do the same. We welcome it.</p>



<p>What we need is the same standard we&rsquo;ve always expected: validation. An AI-assisted finding that&rsquo;s been verified, reproduced, and submitted with a working proof of concept is a great submission. An unvalidated output submitted as-is without reproduction or demonstrated impact is not. This isn&rsquo;t a new standard. It&rsquo;s the same standard we apply to scanner output, static analysis, or any other tool. The human researcher is accountable for the accuracy of the submission.</p>



<p>We&rsquo;d also ask researchers to keep reports concise and structured. A strong report has three things: a <strong>short summary</strong> of the issue, <strong>clear steps to reproduce</strong> with supporting evidence (screenshots, HTTP requests, terminal output), and an <strong>impact statement</strong> explaining what an attacker can actually achieve. That&rsquo;s it. Verbose reports such as multi-page theoretical narratives, restated background context, or AI-generated filler slow down triage because the actual finding gets buried. The clearer and more direct your report, the faster we can act on it.</p>



<p>The tools don&rsquo;t matter. The quality of the work does.</p>



<h2 class="wp-block-heading" id="h-understanding-github-s-security-model-shared-responsibility">Understanding GitHub&rsquo;s security model: Shared responsibility</h2>



<p>One pattern we see frequently deserves its own discussion. Many reports describe scenarios where a user interacts with attacker-controlled content (a malicious repository, a crafted issue, untrusted code) and experiences an undesirable outcome. These reports are often well-written and technically accurate in their observations, but they misunderstand where the security boundary lies.</p>



<p>We invest heavily in systems and teams dedicated to detecting and handling malicious content across the platform, from automated scanning to manual review. That said, GitHub operates on a <strong>shared responsibility model</strong>. Users are responsible for:</p>



<ul class="wp-block-list">
<li><strong>Choosing which repositories, issues, and code they trust.</strong> GitHub hosts over 600 million repositories. Not all of them are benign. Users are expected to exercise judgment about what they interact with.</li>



<li><strong>Reviewing content before executing or interacting with it.</strong> This applies to code, scripts, workflows, and any other executable content.</li>



<li><strong>Understanding that cloning a repository means choosing to trust that code.</strong> Git hooks, build scripts, and other repository-level automation execute because the user chose to check out that repository.</li>



<li><strong>Configuring their own environment securely.</strong> Token management, credential storage, and local security settings are the user&rsquo;s responsibility.</li>
</ul>



<p>When an &ldquo;attack&rdquo; requires the victim to actively seek out and engage with attacker-controlled content (cloning a malicious repo, asking an AI tool to analyze untrusted code, opening a crafted file), the security boundary is the user&rsquo;s decision to trust that content. These scenarios generally don&rsquo;t represent a bypass of GitHub&rsquo;s security controls.</p>



<h3 class="wp-block-heading" id="h-common-examples">Common examples</h3>



<p>To help researchers calibrate, here are patterns we see regularly that fall under shared responsibility:</p>



<figure class="wp-block-table"><table class="has-fixed-layout"><thead><tr><th><strong>Scenario</strong></th><th><strong>Why it&rsquo;s shared responsibility</strong></th></tr></thead><tbody><tr><td>Prompt injection via content the user chose to feed to an AI tool</td><td>The user decided to trust that content</td></tr><tr><td>Git hooks or filters executing code in a repo the user checked out</td><td>This is how Git works by design</td></tr><tr><td>Malicious content in a repository the user cloned</td><td>Cloning is an act of trust</td></tr><tr><td>LLM producing unexpected output when processing untrusted input</td><td>The user chose to provide that input</td></tr></tbody></table></figure>



<p>Research in these areas is still extremely valuable. If you think you&rsquo;ve found a blind spot in our defenses (a way to bypass an actual security control that affects users <em>without</em> requiring them to actively trust malicious content), that&rsquo;s exactly what we want to hear about. Those findings are some of the most impactful submissions we receive. And if you come across content that violates our Terms of Service, please <a href="https://docs.github.com/en/communities/maintaining-your-safety-on-github/reporting-abuse-or-spam">report it</a>.</p>



<h2 class="wp-block-heading" id="h-what-this-means-for-researchers">What this means for researchers</h2>



<p><strong>If you&rsquo;re already submitting quality research</strong>, thank you. Nothing changes for you except faster response times as we reduce queue noise.</p>



<p><strong>If you&rsquo;re newer to bug bounty</strong>, welcome! Take a few minutes to read our scope, review the ineligible list, and invest in a working proof of concept before submitting. Quality submissions from new researchers are always valued and appreciated.</p>



<p><strong>If you&rsquo;ve been prioritizing volume</strong>, we&rsquo;d encourage a shift toward depth. One well-researched, validated finding is worth more than 10 speculative ones, both in bounty payout and reputation. The researchers who earn the most from our program are the ones who go deep.</p>



<h2 class="wp-block-heading" id="h-changes-to-how-we-reward-low-risk-findings">Changes to how we reward low-risk findings</h2>



<p>Not every valid submission represents a meaningful security risk. Some reports identify hardening opportunities or documentation gaps that, while not exploitable, still lead to improvements we choose to make. We appreciate that work.</p>



<p>Going forward, we&rsquo;re updating how we handle these cases. Submissions that don&rsquo;t demonstrate significant security impact but do result in a code or documentation fix will be recognized with <strong>GitHub swag</strong> rather than a bounty payout. This lets us acknowledge the contribution while focusing our bounty resources on the findings that have the greatest impact on platform security.</p>



<p>We&rsquo;d rather see researchers invest their time in deeper, high-impact research and be compensated accordingly than optimize for volume on low-risk findings.</p>



<h2 class="wp-block-heading" id="h-looking-ahead">Looking ahead</h2>



<p>We&rsquo;re committed to making GitHub&rsquo;s bug bounty program one of the best in the industry, for researchers and for the security of the platform. That means faster triage, clearer communication, and ensuring that valid findings get the attention and compensation they deserve. Raising quality standards is part of that investment.</p>



<p>Security researchers make GitHub safer for every developer who depends on it. That work matters, and we don&rsquo;t take it for granted.</p>



<p>Happy hacking! &#128640;</p>

<p>The post <a href="https://github.blog/news-insights/company-news/github-availability-report-april-2026/">GitHub availability report: April 2026</a> appeared first on <a href="https://github.blog">The GitHub Blog</a>.</p>

<p>In April, we experienced 10 incidents that resulted in degraded performance across GitHub services.</p>



<p>To increase transparency, at the end of April, we released a <a href="https://github.blog/news-insights/company-news/an-update-on-github-availability/">blog post</a> covering major incidents on April 23 and April 27. We have also taken steps to <a href="https://github.blog/news-insights/company-news/bringing-more-transparency-to-githubs-status-page/">bring more detail to the GitHub status page.</a></p>



<p>Thank you for your patience as we work through near-term and long-term investments we’re making.</p>



<p><strong>April 01 15:02 UTC (lasting 8 hours and 43 minutes)</strong></p>



<p>On April 1, 2026, between 14:40 and 17:00 UTC, GitHub&#8217;s code search service was fully unavailable; 100% of search queries failed. Service was restored in a degraded state by 17:00 UTC with temporarily stale results, and fully recovered with current data by 23:45 UTC.</p>



<p>During the 2 hour and 20 minute period of full unavailability, 100% of code search requests failed. After initial recovery at 17:00 UTC, search returned results, but they did not reflect repository changes made after approximately 07:00 UTC that day. Full indexing caught up by 23:45 UTC.</p>



<p>During a routine infrastructure upgrade to the messaging system supporting code search, an automated change was applied too aggressively, causing a coordination failure between internal services. This halted search indexing, and search results began going stale. While the team worked to recover the messaging infrastructure, an unintended service deployment cleared internal routing state, escalating the staleness issue into a complete outage.</p>



<p>We restored the messaging infrastructure through a controlled restart, reestablishing coordination between services. We then reset the search index to a point in time before the disruption. No repository data was lost—the search index is a secondary index derived from Git repositories, which were completely unaffected. Once re-indexing completed, all search results reflected the current state of repositories.</p>



<p>We are adding more gradual upgrades with better health checks, so problems are caught before they cascade, deployment safeguards to prevent unintended changes during active incidents, faster recovery tooling to reduce time to restore service and better traffic isolation to prevent cascading impact from unexpected traffic spikes during outages.</p>



<p><strong>April 01 16:06 UTC (lasting 4 minutes)</strong></p>



<p>On April 1, 2026, between 15:34 UTC and 16:02 UTC, our audit log service lost connectivity to its backing data store due to a failed credential rotation. During this 28-minute window, audit log history was unavailable via both the API and web UI. This resulted in 5xx errors for 4,297 API actors and 127 github.com users. Additionally, events created during this window were delayed by up to 29 minutes in github.com and event streaming. No audit log events were lost; all audit log events were ultimately written and streamed successfully. Customers using GitHub Enterprise Cloud with data residency were not impacted by this incident.</p>



<p>We were alerted to the infrastructure failure at 15:40 UTC—six minutes after onset—and resolved the issue by recycling the affected environment, restoring full service by 16:02 UTC. As a result of the incident, we completed several follow-up actions to reduce the risk of recurrence and improve detection. The team updated and strengthened the credential rotation process to improve resiliency and help prevent similar failures in the future. In parallel, we enhanced our monitoring configuration, including making paging thresholds more sensitive to improve detection speed and operator visibility into similar issues going forward.</p>



<p><strong>April 09 09:50 UTC (lasting 25 minutes) and 16:20 UTC (lasting 4 hours and 16 minutes)</strong></p>



<p>On April 9, 2026, we had two incidents, between 09:05 UTC and 19:05 UTC and between 16:05 UTC and 20:36 UTC, where the Copilot coding agent service was degraded and users experienced significant delays starting new agent sessions. Approximately 84% of new agent session requests were delayed across four separate outage waves, with queue wait times peaking at 54 minutes compared to a normal baseline of 15–40 seconds. On average, the error rate was 83.9% and peaked at 97.5% of requests to the service. Approximately 22,700 workflow creations were delayed or failed during the incident.</p>



<p>This was due to a bug in our rate limiting logic that incorrectly applied a rate limit globally across all users, rather than scoping it to the individual installation that triggered the limit. A contributing factor was a surge in API traffic from a client update that increased requests to an internal endpoint by 3–4x, which accelerated rate limit exhaustion. The second similar incident was also caused due to an internal service exceeding API rate limits, compounded by a caching bug that persisted the rate-limited state beyond the actual rate limit window, causing recurring outage.</p>



<p>Our team detected the issue within 15 minutes and began investigating immediately. We mitigated the incident by disabling the faulty rate limit caching mechanism via feature flag and updating our service to use per-installation credentials for API calls, ensuring rate limits are correctly scoped to individual installations. Service was fully restored by 20:36 UTC. Jobs that were delayed during the incident were queued and processed once service resumed.</p>



<p>We have since added automated monitoring and alerting to detect this failure mode proactively, deployed fixes to reduce unnecessary API traffic through caching improvements, and are continuing work to further isolate rate limit scoping across client types to prevent similar issues in the future.</p>



<p><strong>April 13 19:56 UTC (lasting 39 minutes)</strong></p>



<p>On April 13, 2026, between 18:53 UTC and 20:30 UTC, the GitHub Pages service experienced elevated error rates. On average, the error rate was 10.58% and peaked at 12.77% of requests to the service, resulting in approximately 17.5 million failed requests returning HTTP 500 errors.</p>



<p>This incident was due to an automated DNS management tool erroneously deleting a DNS record for a GitHub Pages backend storage host after its upstream data source intermittently failed to return the record, causing the tool to treat it as stale and remove it. As cached copies of the record expired across our systems, GitHub Pages servers were no longer able to reach the affected storage host, resulting in errors for a portion of requests.</p>



<p>Once the issue was identified, our team quickly traced it to the missing DNS record and re-created it. Service returned to normal by 20:30 UTC, and the incident was fully resolved by 20:35 UTC. We recognize that detection took longer than we would have liked—approximately 53 minutes—due to the gradual nature of the error increase and a gap in our alerting for this type of failure.</p>



<p>We are making three key improvements to prevent similar issues in the future. We are implementing availability-zone-tolerant routing in the GitHub Pages frontend so that an unresolvable backend host triggers failover to healthy hosts rather than returning errors, adding safeguards to prevent automated deletion of DNS records owned by other systems, and improving logging and alerting for DNS resolution failures in the GitHub Pages serving path.</p>



<p><strong>April 16 15:06 UTC (lasting 3 hours and 22 minutes)</strong></p>



<p>On April 16, 2026, between 09:30 UTC and 17:15 UTC, users experienced failures when attempting to connect to GitHub Codespaces via the VS Code editor. During this time, approximately 40% of codespace start operations failed. Users connecting via SSH were not impacted.</p>



<p>The issue was caused by failures in an upstream service that prevented the VS Code Server from being retrieved during codespace startup. The impact was mitigated by implementing a workaround to use an alternative download path when the primary endpoint is degraded. In parallel, we coordinated with the upstream dependency team to address the root cause of the download failure.</p>



<p>We are improving our fallback mechanism to reduce the impact of similar upstream failures in the future and streamlining processes to accelerate deployment of similar changes in the future.</p>



<p><strong>April 20 13:28 UTC (lasting 15 hours and 36 minutes)</strong></p>



<p>On April 20, 2026, between 10:28 UTC and 15:04 UTC, GitHub experienced degraded service for code scanning default setup, code quality, and project boards. Repair of affected project boards additionally lasted until 05:04 UTC on April 21.</p>



<p>During this time, code scanning default setup and code quality analyses were not triggered on newly opened pull requests. Additionally, newly created issues were not appearing on project boards.</p>



<p>The cause was a serialization error that prevented proper triggering of code scanning, code quality analyses, and project board updates.</p>



<p>We identified the issue within ~40 minutes and mitigated the issue by deploying a fix, restoring event publishing for code scanning and code quality. For project boards, an additional code change was deployed to update event consumers, followed by a reindex of affected project items.</p>



<p>We are working to prevent recurrence by strengthening our schema validations and improving monitoring for drops in publishing on critical topics. In addition, we are auditing other parts of our system to ensure no similar limitations exist elsewhere.</p>



<p><strong>April 22 15:35 UTC (lasting 3 hours and 43 minutes)</strong></p>



<p>On April 22, 2026, between 15:16 UTC and 19:18 UTC, users experienced errors when interacting with Copilot Chat on github.com and Copilot Cloud Agent. During this time, users were unable to use Copilot Chat or Copilot Cloud Agent. Copilot Memory (in preview) was not available to Copilot agent sessions during this time.</p>



<p>The issue was caused by an infrastructure configuration change that resulted in connectivity issues with our databases. The team identified the cause and restored connectivity to the database. Copilot Chat and Cloud Agent for github.com were restored by 18:16 UTC. Remaining regional deployments were restored incrementally, with full resolution at 19:18 UTC.</p>



<p>We have taken steps to prevent similar infrastructure changes from causing these kinds of database operations in the future.</p>



<p><strong>April 23 16:12 UTC (lasting 1 hour and 18 minutes)</strong></p>



<p>On April 23, 2026, between 16:03 and 17:30 UTC, users experienced elevated error rates and degraded performance across GitHub Copilot, Webhooks, Git Operations, GitHub Actions, Migrations, and Deployments. Approximately 5–7% of overall traffic was affected during the 1 hour and 27 minute impact window. For Copilot, ~7% of AI model requests failed, ~10% of Copilot cloud agent sessions were affected, and ~9% of Copilot Insights dashboard requests returned errors. For Webhooks, ~0.35% of API requests returned errors at peak, with up to 10% of traffic experiencing elevated latency (>3 seconds). Git Operations averaged 1.25% errors over the incident duration, with a peak of 2.07%. GitHub Actions saw workflow run status updates experienced delays of up to ~8 seconds. For Migrations, 0.88% of active repository migrations failed, and 79% saw elevated durations. Deployments were temporarily blocked during the incident window.</p>



<p>Our DNS infrastructure in one datacenter entered a degraded state and began intermittently failing to resolve service addresses. This caused a cascading impact: services that depend on name resolution to communicate with internal APIs, external providers, and storage systems all experienced errors simultaneously.</p>



<p>The root cause was a recently introduced traffic-balancing mechanism that had been rolled out progressively to support our growth. Under a specific load pattern, this mechanism caused DNS resolvers to begin failing. Existing DNS caching provided partial protection. Services with recently cached entries continued operating normally, which is why overall impact was limited to approximately 5–7% of traffic rather than a complete outage.</p>



<p>After an initial configuration rollback did not resolve the issue, we restarted the affected DNS infrastructure. Services began recovering within minutes, and all returned to normal by 17:30 UTC. No data was lost; all repositories, databases, and workflow data were unaffected.</p>



<p>To prevent incidents like this in the future we are improving our DNS infrastructure resilience to prevent single-datacenter failures from cascading across services, implementing safer rollout and validation with a dedicated environment to test infrastructure changes against production-like traffic, investing in faster automated detection and recovery with self-healing mechanisms for DNS resolution failures and reducing blast radius by reviewing service dependencies on shared infrastructure components.</p>



<p><strong>April 27 16:31 UTC (lasting 6 hours and 15 minutes)</strong></p>



<p>On April 27, 2026, between 16:15 UTC and 22:46 UTC, GitHub search services experienced degraded connectivity due to saturation of the load balancing tier deployed in front of our search infrastructure. This resulted in intermittent failures for services relying on our search data including Issues, Pull Requests, Projects, Repositories, Actions, Package Registry and Dependabot Alerts. The impact was varied by search target, with services seeing up to 65% of searches timing out or returning an error between 16:15 UTC and 18:00 UTC.</p>



<p>We detected the drop in search results through our ongoing monitoring and declared an incident at 16:21 UTC. We tracked the incident as mitigated as of 21:33 UTC and monitored the systems until 22:46 UTC when we declared the incident resolved.</p>



<p>The saturation was caused by a large influx of anonymous distributed scraping traffic that was crafted to avoid our public API rate limits. This scraping traffic made up 30% of the day’s total search traffic, but it was concentrated within a four-hour period. The traffic originated from over 600,000 Unique IP addresses, with matching actor information in all requests. Our existing monitoring did not classify the increased scraping as a risk and this dimension of the incident was only discovered while working to mitigate.</p>



<p>To mitigate, we immediately focused on relieving pressure from the load balancers while simultaneously working on scaling the load balancing tier, blocking the anomalous traffic and applying tuning to the balancers to fully resolve the incident.</p>



<p>Looking ahead, we’ve not only scaled the load balancer tier, but applied optimizations to improve our connection handling and re-use to reduce the possibility that a saturation event like this can re-occur. We’ve also added new monitors and controls within the platform to allow us to restrict anonymous traffic to mitigate the impact to our registered users. We are continuing to strengthen our defenses against this type of large-scale automated abuse.</p>
